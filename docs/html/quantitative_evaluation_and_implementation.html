Vintage LLM Project Note

정량평가, talkie-lm.com과의 차이점, 구현 방법

시대중립 프롬프트 전처리 프로젝트를 기계학습개론 과제로 설득력 있게 만들기 위한 평가 설계와 구현 계획을 정리한 문서입니다.

1. 다른 LLM은 품질을 어떻게 정량 평가하는가?

LLM은 하나의 점수로 모든 품질을 평가하기 어렵기 때문에, 보통 능력별 벤치마크를 나누어 사용합니다. 객관식 문제는 accuracy, 코드 생성은 테스트 통과율, 대화 품질은 pairwise win rate나 Elo를 사용합니다.

평가 대상 대표 지표 대표 벤치마크 / 방식 의미
지식·추론 Accuracy MMLU, GPQA 객관식 또는 정답이 있는 문제를 몇 개 맞혔는지 측정
수학 정답률, pass rate GSM8K, MATH, AIME 계산과 추론을 거쳐 최종 정답을 맞히는지 평가
코딩 pass@k, 테스트 통과율 HumanEval, SWE-bench 생성한 코드가 unit test 또는 실제 issue 테스트를 통과하는지 평가
대화 품질 Win Rate, Elo, Judge Score Chatbot Arena, MT-Bench 두 답변 중 더 나은 답변을 사람이거나 judge LLM이 선택
사실성·환각 Truthfulness, Faithfulness TruthfulQA 그럴듯하지만 틀린 답변을 얼마나 줄이는지 평가
RAG 시스템 Context Recall, Faithfulness, Answer Relevance RAGAS 검색된 문서가 충분한지, 답변이 문서에 충실한지 측정
이 프로젝트에 적용할 핵심: 자유 생성 답변의 품질은 하나의 정답과 비교하기 어렵기 때문에, baseline 답변과 proposed 답변을 익명으로 비교하는 Pairwise Win Rate를 메인 지표로 사용합니다.

2. 본 프로젝트의 정량 평가 설계

평가는 두 층으로 나눕니다. 첫 번째는 전처리기가 질문을 잘 바꿨는지 평가하고, 두 번째는 전처리 후 vintage LLM의 답변 품질이 실제로 향상됐는지 평가합니다.

전처리기 자체 평가

  • Anachronism Removal Rate: 제거된 현대 용어 수 / 원문 현대 용어 수
  • Rewrite Success Rate: 검증기를 통과한 변환문 수 / 전체 질문 수
  • Semantic Preservation Score: 원문 질문 의도 보존 정도, 1~5점
  • Leakage Score: 변환문이 답을 미리 알려준 정도, 낮을수록 좋음

최종 답변 품질 평가

  • Answer Relevance: 답변이 질문에 맞는지, 1~5점
  • Comprehension Score: 모델이 핵심 개념을 이해했는지, 1~5점
  • Task Success Rate: 기준 점수 이상 답변 비율
  • Pairwise Win Rate: Proposed가 Baseline보다 낫다고 선택된 비율
Baseline: 현대어 질문 → talkie-1930-13b-it → baseline answer Proposed: 현대어 질문 → 시대중립 전처리기 → talkie-1930-13b-it → proposed answer Pairwise Win Rate 예시: 총 30개 질문 Proposed 승리 21개, Baseline 승리 6개, 무승부 3개 Proposed Win Rate = 21 / 30 = 70.0%

3. talkie-lm.com 소개 글과의 차이점

이 프로젝트는 talkie 소개 글의 실험을 그대로 반복하지 않습니다. talkie 소개 글은 vintage LLM의 성능을 분석하기 위해 시대착오 질문을 탐지하고 필터링하는 데 초점이 있습니다. 반면 본 프로젝트는 그런 질문을 버리지 않고, vintage LLM이 이해할 수 있는 입력으로 재작성하는 서비스를 만듭니다.

구분 talkie-lm.com 소개 글 본 프로젝트
핵심 목적 1930년 기반 LLM과 현대 LLM의 성능·특성 비교 현대어 질문을 vintage LLM에 맞게 변환하는 입력 전처리 서비스 구현
시대착오 질문 처리 시대착오 질문을 탐지하고 평가에서 제외하거나 별도 분석 시대착오 질문을 시대중립 표현으로 재작성하여 사용
입력 데이터 기존 벤치마크 중심 현대 개념이 포함된 질문 20~50개 직접 제작
산출물 모델 소개, 벤치마크 결과, 분석 Detector, Rewriter, Validator, Evaluator로 구성된 파이프라인
연구 질문 1930년 모델은 현대 모델과 비교해 어떤 차이를 보이는가? 시대착오 질문을 재작성하면 vintage LLM의 응답 품질이 향상되는가?

차별화된 메인 질문

시대착오 질문을 단순히 제외하지 않고, 답을 유출하지 않는 시대중립 표현으로 재작성하면 vintage LLM의 활용 가능성과 응답 품질을 높일 수 있는가?

4. 구현 방법

구현은 Python 기반 실험 파이프라인으로 시작하고, 가능하면 Streamlit 같은 간단한 UI를 붙여 발표에서 시연 가능한 서비스 형태로 만듭니다.

1

현대 개념 포함 질문 데이터셋 제작

2

시대착오 개념 탐지기 구현

3

시대중립 재작성기 구현

4

검증기와 자가수정 루프 구현

5

Baseline/Proposed 비교 평가

Detector

직접 만든 현대어 사전과 LLM 기반 탐지를 함께 사용합니다. 명확한 현대 용어는 사전으로 빠르게 잡고, 암묵적인 시대착오 개념은 LLM이 보완합니다.

Rewriter

현대 용어를 직접 쓰지 않고 기능적 설명으로 바꿉니다. 예를 들어 LLM은 “많은 글을 읽고 다음 단어를 예측하도록 훈련된 자동 문장 생성 기계”로 변환합니다.

Validator

변환문에 현대 용어가 남았는지, 원래 질문의 의미가 유지됐는지, 답을 과도하게 알려주지는 않았는지 검사합니다.

파일 구조 예시

data/questions.jsonl src/detector.py src/rewriter.py src/validator.py src/run_talkie.py src/evaluator.py src/run_experiment.py results/baseline_outputs.jsonl results/proposed_outputs.jsonl results/evaluation.csv app.py

실험 루프 예시

for question in dataset: terms = detect_anachronisms(question) rewritten = rewrite_era_neutral(question, terms) verdict = validate_rewrite(question, rewritten) if verdict.failed: rewritten = revise_rewrite(question, rewritten, verdict) baseline_answer = ask_talkie(question) proposed_answer = ask_talkie(rewritten) score = evaluate_pairwise(question, baseline_answer, proposed_answer)

현실적 리스크

talkie-1930-13b-it는 13B 모델이므로 GPU VRAM 요구량이 큽니다. 로컬 실행이 어렵다면 전체 데이터셋은 전처리 품질과 LLM-as-a-Judge 평가 중심으로 진행하고, talkie 직접 실행은 가능한 샘플 수만 수행하는 방식을 권장합니다.
