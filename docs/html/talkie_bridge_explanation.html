TalkieBridge

Era-Neutral Prompt Generator 설명서

이 문서는 저장소의 현재 구현인 src/talkie_bridge/를 기준으로, TalkieBridge가 무엇을 만들었고 어떤 실험 산출물을 내는지 빠르게 이해하기 위한 HTML 설명서입니다.

요약 파이프라인 데이터셋 Leakage 방지 평가 조건 Autoencoder Validator Manual CSV 산출물 테스트 실행 순서

1. 한 줄 요약

TalkieBridge는 Talkie-1930을 학습시키는 프로젝트가 아닙니다. 현대어 객관식 질문을 Talkie-1930이 이해하기 쉬운 시대중립 질문으로 바꾸고, 그 rewrite가 downstream 성능에 도움이 되는지 평가하는 전처리 및 실험 파이프라인입니다.

기준 문서는 PROJECT_PROPOSAL.md이고, 현재 구현의 핵심 위치는 src/talkie_bridge/입니다. prototypes/는 참고용, src/talkie/는 원래 Talkie reference package입니다.

2. 전체 파이프라인

실제 코드 흐름은 아래처럼 이어집니다.

1. Datasetdata/generated_questions.jsonl을 읽고 split을 정합니다.
2. Resourcetrain annotation 또는 explicit dictionary로 generation resource를 만듭니다.
3. Detector현대 용어 alias를 dictionary 기반으로 탐지합니다.
4. Mapper탐지된 용어를 primitive id와 phrase로 연결합니다.
5. Rewriter5가지 조건의 질문을 생성합니다.
6. Validator금지어, primitive, leakage, 길이, 선택지 복사를 검사합니다.
7. Prompt같은 choices/gold answer로 MCQ prompt를 만듭니다.
8. Report수동 Talkie 응답을 평가하고 CSV/Markdown/HTML 산출물을 냅니다.
dataset
  -> detector
  -> primitive mapper
  -> rewrite conditions
  -> validator / deterministic repair
  -> prompt builder
  -> manual Talkie collection
  -> evaluation metrics
  -> report

3. 데이터셋 형식

기본 입력은 data/generated_questions.jsonl입니다. 한 줄에 한 문항 JSON object를 넣습니다.

필드의미
id안정적인 문항 ID. 예: q001
domain문항 도메인. 예: AI_Computing, Medicine_Biology
original_question현대어 원본 질문 stem
choicesA, B, C, D를 가진 JSON object
gold_answer정답 label. A/B/C/D 중 하나
gold_anachronism_terms탐지되어야 하는 현대 용어 목록
forbidden_termsrewrite 후 남으면 안 되는 표현 목록
required_primitives최종 rewrite가 의미상 보존해야 하는 primitive id
primitive_phrase현대 용어를 설명하는 시대중립 기능 문구
human_validated사람이 검수했으면 true
split선택 필드. train/dev/test; 없으면 deterministic 70/15/15 split
 
{"id":"q001","domain":"AI_Computing","original_question":"Why is RAG useful for reducing LLM hallucination?","choices":{"A":"It checks relevant records before writing, so the reply is less likely to rest only on memory.","B":"It improves the result only by making the message longer.","C":"It works mainly by changing the names of answer choices.","D":"It guarantees a correct answer without checking evidence."},"gold_answer":"A","gold_anachronism_terms":["RAG","LLM hallucination"],"forbidden_terms":["RAG","retrieval augmented generation","LLM","large language model","hallucination"],"required_primitives":["record_lookup_before_reply","unsupported_automatic_text"],"primitive_phrase":"first searches a store of relevant records before composing a reply, so unsupported statements from an automatic writing system can be reduced","human_validated":true,"split":"test"}

4. Generation Resource와 Leakage 방지

기본 원칙

기본 generation resource는 train split annotation으로만 만듭니다. dev/test 문항의 gold annotation이 rewrite 생성에 들어가지 않도록 분리합니다.

train-only post-hoc validation

명시 dictionary

predeclared dictionary는 --concept-dictionary-json과 --primitive-dictionary-json을 둘 다 넘겼을 때만 사용합니다. 파일 이름으로 자동 로드하지 않습니다.

explicit only no auto-load
  • forbidden_terms, required_primitives, primitive_phrase는 eval item에서 post-hoc validation/metrics용입니다.
  • Repair loop는 eval item의 gold primitive로 제어되지 않고, detector/mapper가 만든 primitive만 보강합니다.
  • mock_*dictionary.json은 final evaluation에서 기본 차단됩니다.

5. 다섯 가지 평가 조건

모든 조건은 같은 문항, 같은 choices, 같은 gold answer를 씁니다. 바뀌는 것은 question text뿐입니다.

조건무엇을 평가하나
raw현대어 원본 질문을 그대로 넣는 baseline
rule_onlydictionary/primitive 기반 치환만 쓴 baseline
length_controlledproposed와 길이를 맞춰 prompt 길이 효과를 통제
proposeddetector + primitive bottleneck + validator/repair를 모두 사용
proposed_no_validatorvalidator/repair 효과를 보기 위한 ablation

6. Autoencoder / Learned Component

중요: autoencoder.py는 자연어 문장을 생성하는 full generator가 아닙니다. dependency-free learned primitive selector, 즉 primitive bottleneck입니다.

입력 질문의 bag-of-words를 낮은 차원 latent representation으로 압축하고, primitive phrase token을 복원하도록 학습합니다. 이후 원문에 맞는 primitive 후보를 점수화해 proposed condition에 보조로 추가합니다.

  • train split으로 학습합니다.
  • dev split primitive F1로 config를 선택합니다.
  • train 4개 미만 또는 dev 없음이면 deterministic fallback합니다.
  • fallback 사유와 선택 config는 cache/primitive_autoencoder_selection.json에 기록됩니다.

보고서에서는 이 component를 "문장 생성 모델"이 아니라 "rule mapper를 보조하는 작은 learned primitive selector"로 해석해야 합니다.

7. Detector와 Primitive Mapper

Dictionary Detector

alias를 longest-first regex로 매칭해서 현대 용어 span, canonical term, primitive id를 찾습니다. 실제 rewrite를 구동하는 주 detector입니다.

Bernoulli Naive Bayes

train 질문 token과 modern-term token으로 학습해 token별 modern score를 냅니다. 현재는 diagnostic 기록용이며 dictionary detection을 대체하지 않습니다.

Primitive mapper는 detector가 찾은 term의 primitive_id를 모아 primitive_dictionary에서 primitive_phrase를 꺼냅니다.

8. Validator와 Repair Loop

validator.py는 rewrite가 실험적으로 공정한지 검사합니다.

검사의미
Forbidden terms remaining현대 용어가 rewrite에 남았는지 확인
Required primitive recallprimitive phrase keyword가 충분히 보존됐는지 확인
Choice-copying risk선택지 문장을 과도하게 베꼈는지 확인
Answer-hint risk정답 선택지와 keyword overlap이 높은지 확인
Leakage phrasescorrect answer, option A 같은 직접 힌트 확인
Length ratio원문 대비 너무 짧거나 긴 rewrite 차단

Repair loop는 최대 2번 deterministic하게 실행됩니다. 남은 detected term 제거, leakage phrase 삭제, generation primitive phrase 보강, 일부 choice-copying/hint 표현 완화를 합니다. eval gold annotation을 새로 읽어 정답 힌트를 삽입하지 않습니다.

9. Manual CSV 평가 흐름

최종 재현성 경로는 Talkie 웹 응답을 사람이 CSV에 채우는 manual mode입니다. 

python -m talkie_bridge.cli init-mock-data --force
python -m talkie_bridge.cli prepare-manual
python -m talkie_bridge.cli evaluate-manual --manual-response-csv input_data/manual_talkie_input_sheet.csv
python -m talkie_bridge.cli demo --item-id q001 --out results/demo.html
Manual integrity: prepare-manual이 만든 prompt_hash가 평가 시 다시 확인됩니다. hash mismatch와 blank response는 기본 실패입니다.

필요한 경우에만 diagnostic 목적으로 --allow-hash-mismatch 또는 --allow-missing-responses를 씁니다.

11. 테스트

tests/는 구현의 핵심 안전장치를 확인합니다.

  • test_detector_mapper.py: alias detection과 primitive mapping 검증
  • test_validator.py: forbidden term, leakage phrase, deterministic repair 검증
  • test_prompting_metrics.py: strict answer parsing, invalid rate, paired McNemar 검증
  • test_cli_pipeline.py: rewrite/manual/evaluate/demo 산출물 생성, blank response 실패, dictionary 자동 로딩 방지
  • test_autoencoder.py: train/dev 학습, eval annotation leakage 방지, gold primitive가 repair를 제어하지 않는지 검증

현재 확인한 환경에서는 pytest가 설치되어 있지 않아 python -m pytest -q는 실행되지 않았습니다.

12. 실제 실험 전 남은 작업

  • 100~150개 human_validated=true 문항 작성
  • independent predeclared dictionary 작성 또는 train-only resource 사용 결정
  • Talkie 웹에서 manual response 수집
  • 최종 evaluate-manual 실행
  • raw, rule_only, length_controlled, proposed, proposed_no_validator 비교 해석

최종 실험 실행 순서

실제 validated dataset을 준비한 뒤 아래 순서로 실행합니다.

python -m pip install -e .

# data/generated_questions.jsonl에 최종 검수 데이터 배치
python -m talkie_bridge.cli prepare-manual --eval-split test

# input_data/manual_talkie_input_sheet.csv에 Talkie 응답을 채운 뒤
python -m talkie_bridge.cli evaluate-manual --eval-split test --manual-response-csv input_data/manual_talkie_input_sheet.csv

# 선택: 한 문항 demo HTML 생성
python -m talkie_bridge.cli demo --item-id q001 --out results/demo.html
해석 포인트: proposed가 raw보다 높아도 충분하지 않습니다. rule_only, length_controlled, proposed_no_validator와 함께 봐야 성능 향상이 단순 길이 증가나 정답 힌트 때문이 아닌지 판단할 수 있습니다.
