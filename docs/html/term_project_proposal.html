1. 연구 배경과 문제 정의

배경

talkie-1930 계열 모델은 1931년 이전 영어 텍스트를 중심으로 학습된 vintage language model입니다. 이 모델은 현대 웹 지식이 거의 없는 환경에서 언어모델의 추론과 일반화 능력을 관찰할 수 있다는 점에서 흥미롭습니다.

문제

사용자가 현대어로 질문하면 모델이 질문의 핵심을 이해하지 못할 수 있습니다. 예를 들어 LLM, RAG, 스마트폰, 인터넷 같은 표현은 1930년 이전 지식만으로는 직접 이해하기 어렵습니다.

따라서 본 프로젝트는 현대 개념이 포함된 입력을 시대중립적 설명으로 변환하는 전처리 시스템을 만들고, 이 전처리가 vintage LLM의 응답 품질을 실제로 개선하는지 검증합니다.

3. 제안 방법론

Anachronism Detection

입력 문장에서 1930년 이후 등장한 기술, 사건, 기관, 약어, 현대 관용어를 탐지합니다. 탐지 결과는 변환 근거와 평가 로그로 저장합니다.

Era-Neutral Rewriting

현대 용어를 직접 사용하지 않고, 답을 알려주지 않는 최소한의 정의로 질문을 다시 작성합니다. 핵심은 문제의 추론 과제는 유지하면서 배경 지식 격차만 줄이는 것입니다.

Self-Refinement

변환 결과를 다시 검사하여 남아 있는 현대 개념을 찾고, 실패하면 재작성합니다. 이는 test-time scaling 또는 self-feedback 전략으로 볼 수 있습니다.

Original: LLM hallucination을 줄이려면 RAG가 왜 도움이 돼? Rewritten: 어떤 자동 문장 생성 기계가 많은 글을 읽고 문장을 이어 쓰도록 훈련되었다고 하자. 이 기계는 때때로 실제 근거가 없는 말을 그럴듯하게 만들어낸다. 답하기 전에 별도의 문서 창고에서 관련 문서를 찾아 참고하게 하면, 왜 그런 오류가 줄어들 수 있는가?

5. LLM 품질 정량 평가 방식

LLM의 품질은 하나의 점수로만 평가하기 어렵기 때문에, 일반적으로 지식, 추론, 코딩, 대화 선호도, 사실성, RAG 품질처럼 능력별 지표를 나누어 정량 평가합니다. 본 프로젝트도 이 관례를 따라 전처리 품질과 최종 답변 품질을 분리해서 측정합니다.

평가 대상 대표 지표 대표 벤치마크 / 방식 본 프로젝트 적용 방식
지식·추론 Accuracy MMLU, GPQA 질문 의도에 맞는 답변을 했는지 1~5점 또는 성공/실패로 평가
코딩 pass@k, 테스트 통과율 HumanEval, SWE-bench 정답이 명확한 질문은 expected answer와 비교 가능
대화 품질 Win Rate, Elo, Judge Score Chatbot Arena, MT-Bench Baseline 답변과 Proposed 답변을 익명 비교하여 Pairwise Win Rate 계산
사실성·환각 Truthfulness, Faithfulness TruthfulQA, factuality 평가 전처리문이 답을 유출하지 않았는지, 모델 답변이 근거 없는 내용을 만드는지 평가
RAG 시스템 Context Recall, Faithfulness, Answer Relevance RAGAS Answer Relevance와 Semantic Preservation을 LLM-as-a-Judge로 측정

전처리기 정량 평가

  • Anachronism Removal Rate = 제거된 현대 용어 수 / 원문 현대 용어 수
  • Rewrite Success Rate = 검증기를 통과한 변환문 수 / 전체 질문 수
  • Semantic Preservation Score = 원문 의미 보존 정도, 1~5점
  • Leakage Score = 답을 미리 알려준 정도, 낮을수록 좋음

최종 답변 정량 평가

  • Answer Relevance = 질문에 맞는 답변인지 1~5점
  • Comprehension Score = 핵심 개념을 이해했는지 1~5점
  • Task Success Rate = 기준 점수 이상 답변 비율
  • Pairwise Win Rate = Proposed가 Baseline보다 낫다고 선택된 비율
Pairwise Win Rate 예시: 총 30개 질문 Proposed 승리: 21개 Baseline 승리: 6개 무승부: 3개 Proposed Win Rate = 21 / 30 = 70.0%

6. talkie-lm.com 소개 글과의 차이점

본 프로젝트는 talkie-lm.com의 실험을 그대로 반복하는 것이 아니라, 그 문제의식을 서비스 형태로 확장합니다. talkie 소개 글은 vintage LLM을 평가하기 위해 시대착오 질문을 필터링하거나 제외하는 방향에 가깝습니다. 반면 본 프로젝트는 시대착오 질문을 버리지 않고, vintage LLM이 이해할 수 있는 시대중립 입력으로 재작성합니다.

구분 talkie-lm.com 소개 글 본 프로젝트
핵심 목적 1930년 기반 LLM과 modern twin의 성능 및 특성 분석 현대 사용자의 질문을 vintage LLM에 맞게 변환하는 입력 전처리 서비스 구현
시대착오 질문 처리 평가 데이터에서 시대착오적 질문을 탐지하고 필터링 시대착오적 질문을 시대중립 표현으로 재작성하여 사용 가능하게 만듦
산출물 모델 소개, 벤치마크 분석, 비교 실험 결과 Detector, Rewriter, Validator, Evaluator로 구성된 전처리 파이프라인
평가 질문 기존 벤치마크 중심 현대 개념이 포함된 직접 제작 질문 20~50개
연구 질문 1930년 모델은 현대 모델과 비교해 어떤 차이를 보이는가? 시대착오 질문을 제거하지 않고 재작성하면 vintage LLM의 응답 품질이 향상되는가?

차별화된 메인 질문

시대착오 질문을 단순히 제외하지 않고, 답을 유출하지 않는 시대중립 표현으로 재작성하면 vintage LLM의 활용 가능성과 응답 품질을 높일 수 있는가?

실행 흐름 예시

for question in dataset: terms = detect_anachronisms(question) rewritten = rewrite_era_neutral(question, terms) verdict = validate_rewrite(question, rewritten) if verdict.failed: rewritten = revise_rewrite(question, rewritten, verdict) baseline_answer = ask_talkie(question) proposed_answer = ask_talkie(rewritten) score = evaluate_pairwise(question, baseline_answer, proposed_answer)

Detector

직접 만든 현대어 사전과 LLM 탐지를 함께 사용합니다. 사전은 명확한 용어를 빠르게 잡고, LLM은 사전에 없는 시대착오 개념을 보완합니다.

Rewriter

탐지된 용어를 기능적 설명으로 바꿉니다. 예를 들어 LLM은 “많은 글을 읽고 다음 단어를 예측하도록 훈련된 자동 문장 생성 기계”로 변환합니다.

Validator

변환문에 현대 용어가 남았는지, 원래 질문의 의도가 유지됐는지, 답을 너무 알려주지는 않았는지 검사합니다.

실행 리스크와 현실적 대안

talkie-1930-13b-it는 GPU VRAM 요구량이 높기 때문에 전체 실험을 로컬에서 수행하기 어려울 수 있습니다. 이 경우에는 전처리 파이프라인과 LLM-as-a-Judge 평가는 전체 데이터셋에 대해 수행하고, talkie 직접 실행은 가능한 샘플 수만큼만 수행하는 방식으로 리스크를 줄입니다.

8. 기대 결과와 의의

기대 결과

현대어 질문을 그대로 입력한 경우보다, 시대중립 전처리를 적용한 경우 vintage LLM의 답변이 더 관련성 있고 일관적으로 나올 것으로 예상합니다.

의의

단순히 LLM에게 질문하는 서비스가 아니라, 모델의 지식 시점과 사용자 입력 사이의 간극을 줄이는 입력 변환 계층을 설계하고 평가한다는 점에서 기계학습 프로젝트로 확장성이 있습니다.

9. 리스크와 대안

리스크

  • talkie-1930-13b-it는 13B 모델이므로 로컬 실행에 높은 GPU VRAM이 필요합니다.
  • 전처리기가 과도한 설명을 넣으면 답을 유도하는 정보 주입이 발생할 수 있습니다.
  • LLM-as-a-Judge 평가는 평가 모델의 편향에 영향을 받을 수 있습니다.

대안

  • talkie 실행이 어려우면 일부 샘플만 실행하고, 나머지는 전처리 품질 평가 중심으로 진행합니다.
  • 변환 규칙을 명시하고 원문·변환문·평가 로그를 모두 저장해 재현성을 확보합니다.
  • 자동 평가와 사람 평가 샘플을 함께 사용해 결과 신뢰도를 높입니다.
